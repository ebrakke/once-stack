Implement Option A as a small Go-native design system.

Each ONCE app remains independently deployed — Notes, Files, or Blog — but they share the same layout primitives, colors, typography, buttons, cards, forms, markdown styles, and rendering helpers.

Recommendation

Use Go for rendering and Tailwind for styling: html/template + embed.FS + a focused pkg/ui package, with a customized Tailwind build for the shared visual system. Avoid frontend UI frameworks, but do not reinvent Tailwind's utility system.

Research findings: idiomatic Go + Tailwind without frontend frameworks

1

Use html/template

It is the standard library choice for server-rendered HTML and performs context-aware escaping. Keep using template.HTML only for sanitized markdown output.

2

Embed templates/assets

embed.FS keeps Docker images self-contained and avoids runtime path issues. Parse at startup and fail fast.

3

Compose named templates

Use {{define}}, {{template}}, and preferably {{block}} for layouts and page content.

4

Share package APIs, not file paths

Create pkg/ui with a narrow renderer API. Apps pass their embedded templates into it.

5

Small pure helper funcs

Register helpers before parsing: date formatting, class joining, dict construction, current-year, app labels. Keep business logic in handlers.

6

Lean into Tailwind

Use Tailwind's theme customization, utilities, forms/typography patterns, and build-time purge. The no-framework constraint should mean no React/Vue/Svelte app, not no CSS tooling.

Proposed architecture

Move shared layout and design primitives into pkg/ui. Keep app-specific pages in each app package, so binaries remain independent and only import the UI code they need.

pkg/ui/ ├── renderer.go # Renderer type, Parse, Render, FuncMap ├── data.go # Page, App, NavItem, Flash types ├── templates/ │ ├── layout.html # Shared document shell and Option A header │ ├── error.html # Shared error page │ └── components.html # Optional named partials for repeated markup ├── styles/ │ └── once.css # Tailwind input: @tailwind + @theme/@layer customizations └── static/ └── once.css # Compiled Tailwind output embedded in Go binaries pkg/notes/ ├── templates.go # Notes-specific renderer setup └── templates/ ├── index.html # Defines "content" or page template ├── view.html └── form.html

Renderer shape

package ui

type App struct {
    Name     string // "Notes"
    BasePath string // usually "/"
}

type Page struct {
    App   App
    Title string
    Error string
    Data  any
}

type Renderer struct { templates *template.Template }

func NewRenderer(appFS fs.FS, appPatterns ...string) (*Renderer, error)
func (r *Renderer) Render(w io.Writer, name string, page Page) error
Key idea: app handlers build typed view models, then wrap them in ui.Page{App: ui.App{Name:"Notes"}, Title:"Notes", Data: notes.IndexData{...}}. Shared templates read common layout fields; page templates read .Data.

Shared UI elements to extract first

Tailwind direction: define Option A as a Tailwind theme/preset plus a few semantic component classes in @layer components. Templates can still use normal Tailwind utilities for one-off layout, while repeated patterns use classes like once-btn, once-card, and once-input.

Implementation task breakdown

Phase 0
decision

Confirm design direction and scope

  1. Approve Option A as the default ONCE visual language.
  2. Decide Tailwind build mechanics. Recommendation: source CSS in pkg/ui/styles/once.css, compile to pkg/ui/static/once.css, and serve the embedded compiled file.
  3. Agree that each app has its own header identity and no cross-app shell.
Phase 1
foundation

Create pkg/ui

  1. Add Tailwind config/input CSS for the Option A theme.
  2. Add a build command, likely bunx @tailwindcss/cli -i pkg/ui/styles/once.css -o pkg/ui/static/once.css --minify.
  3. Add embedded shared templates and compiled CSS.
  4. Add Renderer, Page, App, and small FuncMap.
  5. Add ui.HandleStatic() or ui.StaticHandler() for embedded CSS.
  6. Add unit tests that parse templates and render a tiny fixture page.
Phase 2
notes migration

Migrate Notes to shared UI

  1. Replace Tailwind CDN usage with the shared compiled Tailwind CSS asset.
  2. Update pkg/notes/templates.go to call ui.NewRenderer.
  3. Split TemplateData into page-specific view models if useful: IndexData, ViewData, FormData.
  4. Keep handler render wrappers stable initially: RenderIndex, RenderView, RenderForm, RenderError.
  5. Run go test ./... and manually test create/view/edit/delete.
Phase 3
hardening

Make shared components robust

  1. Add golden-ish rendering tests for shared error and layout output.
  2. Normalize title generation: avoid strings like Title — Notes — Notes.
  3. Document template naming conventions to avoid collisions.
  4. Verify sanitized markdown remains the only source of template.HTML.
Phase 4
reuse

Apply to Files and Blog as they become real apps

  1. Files uses same header, cards/list rows, empty state, forms/buttons.
  2. Blog uses same system with optional article typography variant from Option C.
  3. Extract any repeated app-specific patterns only after the second app needs them.

Risks and guardrails

Template name collisions

All names live in one template tree. Use namespaced templates like ui/layout, ui/button, notes/index.

Over-abstracting too early

Start with Tailwind tokens/classes and layout. Avoid generic template components that make simple HTML hard to read.

Security with raw HTML

Continue to sanitize markdown before assigning template.HTML. Do not add generic "safeHTML" helpers.

Build and asset route conflicts

Keep Tailwind build outputs deterministic and mount compiled assets under a reserved path such as /assets/once/once.css from pkg/server or each app mux.
